# v0.0.2-beta — In progress

## Overview

Ongoing development and refinements for the Lucis compiler and its ecosystem. This release focuses on developer experience improvements and stability, and marks the official transition of the project's identity.

## Global Rename: Lux → Lucis

The project has undergone a massive rebranding and refactoring. **Lux** is now **Lucis** (the genitive of *Lux* in Latin). This "Global Rename Refactor" represents the first step in a larger transition.

### Changes Included

- **Core Branding**: All references to "Lux" in the compiler, documentation, and metadata have been updated to "Lucis".
- **File Extensions**: Source files transitioned from `.lx` to `.lc`.
- **Directory Structure**: Folders and assets (logos, icons) renamed to match the new identity.
- **Grammar & Internals**: Parser and Lexer definitions updated (`LuxParser` -> `LucisParser`).
- **Tooling Migration**: The VS Code extension and LSP have been updated to the `lucis` namespace.

> **Note**: As this was a massive automated rename, some temporary regressions in the LSP and documentation might exist. These will be addressed throughout the 0.0.2-beta cycle.

## What's New

### Breaking: `namespace` Removed — Path-Based `use` Imports

The `namespace` keyword has been removed from the language. Files no longer declare
`namespace X;` — the file's path relative to the project root **is** the module identity.
`use` is now resolved by file path instead of namespace name.

**Migration:**
```lucis
// Before (v0.0.1)
namespace Main;
use stdio::printf;

// After (v0.0.2)
use stdio::printf;
```

- `use lib::math::add` resolves to `lib/math.lc` (searched in project root + `source` paths from `lucis.yaml`).
- The `source` field in `lucis.yaml` is now used as search prefixes for `use` resolution.
- `ProjectScanner` removed — no directory scanning; the compiler uses BFS import resolution
  starting from the entry point. Only files referenced by `use` are opened.
- `NamespaceRegistry` → `ModuleRegistry` (keyed by `/`-separated file paths).
- IR mangling uses module path (e.g., `lib/math` → `lib_math__add`) instead of namespace name.

### Standard Library (stdlib)

- **Initial Release**: Introduced the first modules of the Lucis standard library.
- **`stdio` Module**: Added `printf(format, ...)` — formatted output with variadic arguments and `{s}`, `{d}`, `{l}`, `{f}` / `{f:N}` placeholders. Replaces the previous `write` wrapper.
- **System Installation**: Stdlib `.lc` files are now installed to `${CMAKE_INSTALL_PREFIX}/share/lucis/stdlib` and auto-discovered by the compiler.

### Standard Library Extras

- **`string.ptr()`**: Added `.ptr()` method on string type returning `*void`, enabling raw pointer access to string data.

### LSP Performance

- **Fixed**: C header scanner no longer follows symlinks (`follow_directory_symlink` removed) — previously caused runaway recursive scans into system directories outside the include tree, consuming excessive CPU and memory.
- **Fixed**: `ProjectScanner` now skips `build/`, `.git/`, `.svn/`, `.hg/` directories to avoid polluting the project context with unrelated `.lc` files.
- **Fixed**: `.lucis/` cache directory is now auto-created in the project root on LSP startup, ensuring the header cache persists across sessions instead of re-scanning system headers every time.

### Project Management

- **Enhanced**: `lucis build` and `lucis run` now work without a file argument — they auto-resolve the entrypoint from `lucis.yaml` when run inside a project directory. Config values (opt level, linker flags, etc.) are used as defaults and can be overridden by CLI flags.
- **New**: `lucis.yaml` project configuration file — defines project name, binary output, source directories, build/run/linker settings, and include paths.

- **New**: `ProjectScanner` respects `source` paths from `lucis.yaml` — when configured, only those directories are scanned for `.lc` files, eliminating cross-project contamination.
- **Dependency**: `yaml-cpp` added (0.7+ required) for parsing `lucis.yaml`.
- **Fixed**: `lucis build` now uses `out_dir` from `lucis.yaml` as the output directory for the final binary (not for intermediate artifacts).
- **Fixed**: `lucis build` no longer appends `.out` when `binary` is set in `lucis.yaml` — the configured name is used verbatim.
- **Fixed**: `.lucis/` cache directory is now always created at the project root (where `lucis.yaml` lives), not next to the input file. Intermediate `.o` files go to `.lucis/build/`; the config's `out_dir` controls only the final binary location.
- **Fixed**: `-o, --output` flag was missing from `build --help`; now properly registered and displayed.
- **Enhanced**: Emit modes (`--emit-llvm`, `--emit-asm`, `--emit-bc`, `--emit-obj`) now use a structured `build.emits` configuration in `lucis.yaml`, with `enabled` and `file` fields per emit type. Multiple emits can be enabled simultaneously; the build generates all specified outputs and stops before linking. CLI flags override the config `enabled` status. Path conflicts between emits produce a clear error.

### Compiler Internals: SemanticDB — Unified Declaration Store

The compiler's type system has been fundamentally refactored. The previous architecture duplicated type registries, template maps, and method tables between the Checker and IRGen phases — creating risk of data loss and requiring re-walking of external ASTs for cross-file resolution.

**SemanticDB** replaces all of this with a single, centralized declaration database:

- **Single source of truth**: All types, functions, generics, and methods are stored once and shared across all compiler phases.
- **Polymorphic `Decl` hierarchy**: Replaces the flat `TypeInfo` struct with 11 strongly-typed declaration subtypes (`StructDecl`, `EnumDecl`, `FunctionDecl`, `GenericTemplateDecl`, etc.), eliminating union-like field sharing and making it impossible to access struct fields on an enum.
- **Forward declaration support**: Recursive and cross-file type references use explicit `forwardDeclare()` → `registerType()` two-phase registration, replacing the fragile "skeleton" upgrade pattern.
- **Cross-file resolution without AST re-walking**: The IRGen phase now resolves external types via `syncTypeFromSemanticDB()` — converting Decl data to TypeInfo directly instead of re-visiting the original file's parse tree.
- **Generic template caching**: Template instantiation uses `clone()` + `substituteTypes()` on fully-resolved patterns (no AST re-walking), with automatic deduplication via mangled name caching.
- **Consistency verification**: `verifySemanticDBConsistency()` automatically compares legacy and new registries after each check unit, detecting data loss during the migration period.
- **Incremental build cache**: The SemanticDB is serialized to `.lucis/build/cache/semantic.db` (line-based text format) after successful semantic analysis, enabling future incremental compilation.
- **LSP navigation independence**: LSP go-to-definition now uses stored line/column metadata instead of accessing the parse tree, a step toward full AST-back-pointer removal.

**Scope**: ~2000 lines across 12 files. All 66 compiler tests pass. The legacy `TypeRegistry` remains as a transitional bridge; full removal is planned for a future release.

See `docs/compiler/semantic-db.md` for complete architecture documentation.

### Cross-File Generic Resolution Fixes

Several interconnected bugs in cross-file generic type resolution were fixed across the Checker, IRGen, and LSP layers. The root cause was the same pattern in multiple places: when a type alias like `type DivideResult = Result<int32>` is imported cross-file, its transitive generic dependency (`Result<T>`) was not registered before the alias was resolved.

- **Checker**: `ensureTypeDependencyFromSpec()` was only called for function dependencies, not type alias dependencies. Added the call before `checkTypeAliasDecl()` in both same-namespace and user-import cross-file registration paths.
- **IRGen**: Same gap — `ensureTypeDependencyFromSpec()` was called for functions, structs, and unions, but not before `visitTypeAliasDecl()` in `registerCrossFileSymbols`.
- **HoverProvider**: `inferCatchUnwrapItType()` and `inferExprTypeName()` for `CatchUnwrapExpr` looked up enum declarations by name only, failing to resolve through type aliases. Added `resolveEnumForCatch()` helper that resolves type aliases (local + cross-file) before looking up the underlying enum.
- **CompletionProvider**: Same code duplication as HoverProvider — `inferCatchUnwrapSuccessType()`, `inferCatchUnwrapItType()`, `addEnumVariants()`, and `addEnumWildcardVariants()` all lacked type alias resolution. Applied the same fix to all four.
- **CompletionProvider + DefinitionProvider**: `findSymbol()` in `ModuleRegistry` stores modules keyed by `/`-separated paths (`lib/math`), but LSP `use`-decl parsing built paths with `::` (`lib::math`). Fixed by calling `usePathToModulePath()` before `findSymbol()` in both autocomplete and go-to-definition paths.
- **CompletionProvider**: Enum variant completion (`Type::|`) stripped generic type arguments (`<int32>`) from the enum name before lookup, so `Result<int32>::|` now correctly suggests `Ok` and `Err`.

### Error Handling: `defer` Now Executes Before `throw`

**Fixed**: `defer` statements and auto-cleanup of owned resources previously did not execute on the `throw` path. The `throw` statement used `longjmp` which bypassed normal scope exit, causing resource leaks.

- `visitThrowStmt()` now calls `emitAllCleanups()` before the `lucis_eh_throw` call, ensuring all `defer` statements and auto-drops (owned strings, `vec<T>`, etc.) execute in LIFO order.
- The defer stack is cleared after emission to prevent double-execution when the function scope exits after a `catch` block.
- Works for both handled (`try`/`catch`) and unhandled throws.

### Error Handling: `?` Propagate Operator Enhancements

The `?` (propagate) operator received significant improvements for ergonomics and flexibility:

- **Compatible enums**: `?` now works between different enum types that share the same error payload (e.g., `MyResult<int32>` → `YourResult<int32>` when both use `Error`). Previously required exact type match.
- **Void functions**: `?` is now allowed in functions returning `void` — propagates the error and discards the success value. Supports unit success variants (e.g., `Ok` without payload).
- **`#[error]` attribute**: Custom variant names can be marked as the error variant via `#[error]` prefix, bypassing the hardcoded naming convention (`Err`/`Error`/etc).
  ```lucis
  enum Status<T> {
      #[error]
      Fault(Error),
      Good(T)
  }
  ```
- **Context block**: `?` accepts an optional block `{ ... }` that executes on the error path before propagation — useful for logging, cleanup, or side effects.
  ```lucis
  auto val = load_config() ? { printf("config load failed\n"); };
  ```
- **Fixed**: `resolveExprTypeInfo` had no handler for `PropagateExprContext`, causing `auto x = expr ?;` to allocate wrong LLVM types for struct payloads (e.g., `string` was allocated as `i32`).

### LSP: Cross-File Completions and Enum Variants

- **Fixed**: `findSymbol()` module path conversion — ModuleRegistry stores modules keyed by `/` paths but LSP passed `::` paths. Added `usePathToModulePath()` conversion in both autocomplete and go-to-definition.
- **Fixed**: `addEnumVariants()` and `addEnumWildcardVariants()` now resolve type aliases to find the underlying enum for `Type::|` completions and `use Type::*;` variant injection.
- **Fixed**: Explicit generic type args in enum access (`Result<int32>::|`) now correctly strip `<...>` before looking up variants.
- **Fixed**: Completion and hover for `catch` expressions now resolve type aliases (e.g., `DivideResult` → `Result<int32>`) before finding enum variants.

---

- **Fixed**: `'self' is not a struct or union` error in struct `extend` method compound assignments (`self.pos++`, `self.pos += 1`). `resolveIncrDecrTarget` in IRGen now correctly auto-derefs pointer receivers (e.g., `*Lexer` → `Lexer`) before field lookup, matching the existing auto-deref logic in `visitFieldAccessExpr`.
- **Fixed**: `unknown method` errors on chained struct method calls (`self.peek().isAlpha()`). Two root causes:
  1. `resolveExprTypeInfo` did not auto-deref Pointer receivers, causing struct method type resolution to fail for chained calls through `&self` parameters.
  2. `structMethods_` return type resolution inferred `int8` instead of `char` from the LLVM function signature (both `i8`). The map now stores the declared return `TypeInfo*` alongside the `llvm::Function*` to preserve exact types.
- **Refactored**: `structMethods_` and `staticStructMethods_` maps changed from `map<string, Function*>` to `map<string, MethodEntry>` (holding both `Function*` and return `TypeInfo*`), preventing future type ambiguities for any primitive with identical LLVM representation.

### Literal Suffixes (Rust-style)

- **New**: Integer and float literal suffixes for all primitive types:
  - Integer: `42i8`, `0xFFu8`, `42i16`, `0o755u16`, `42i32`, `0b1010u32`, `42i64`, `42u64`, `42i128`, `0xDEADBEEFu64`, `42isize`, `42usize`, `42iinf`
  - Float: `3.14f32`, `3.14f64`, `3.14f80`, `3.14f128`
  - Works with all literal forms (decimal, hex `0x`, octal `0o`, binary `0b`, float, `.5f32`).
  - Checker resolves the exact type (`int8`, `uint64`, `float32`, etc.) instead of the default `int32`/`float64`.
  - IRGen creates LLVM constants with the correct bit width, bypassing the default widening/truncation logic.

- **New**: Integer literal with float suffix (`4f32` → `float32`) and float literal with integer suffix (`4.2i32` → `int32`, truncated).
  - Tokenized as combined lexer tokens (`SUFFIXED_INT_FLOAT`, `SUFFIXED_FLOAT_INT`).
  - Full checker, IRGen, and LSP support (semantic tokens, hover, completion).

- **LSP Support**: Literal suffixes now properly handled in all LSP features:
  - `SemanticTokensProvider` emits `Number` semantic tokens for all 8 suffixed token types.
  - `HoverProvider` and `CompletionProvider` resolve suffixed literals to their correct type name (e.g., hovering `42u8` shows `uint8`).

### Type System

- **Fixed**: All four compound-assignment checkers (`x += expr`, `p.x += expr`, `p->x += expr`, `*p += expr`) now verify the RHS type is assignable to the LHS type via `isAssignable()`. Previously only checked numeric/integer category — `float32 += 4` (int32 literal) would silently pass the checker and crash during IRGen. Now correctly rejected with a clear type mismatch error. Use `float32 += 4f32` instead.

- **`open(path, flags, mode) → int32`**: Opens a file, returns a file descriptor. POSIX flags, cross-platform via C wrappers with `#ifdef _WIN32`.
- **`read(fd, buf, count) → int32`**: Reads bytes from a file descriptor.
- **`write(fd, buf, count) → int32`**: Writes bytes to a file descriptor.
- **`close(fd) → int32`**: Closes a file descriptor.
- **`lseek(fd, offset, whence) → int64`**: Repositions the file offset.
- **`fsync(fd) → int32`**: Synchronises written data to storage.
- **`unlink(path) → int32`**: Deletes a file from the filesystem.
- **`file_size(fd) → int64`**: Returns the size of an open file via `fstat`.
- **`dup(fd) → int32`**: Duplicates a file descriptor.
- **`dup2(oldfd, newfd) → int32`**: Duplicates to a specific fd (for redirection).
- Backed by `src/builtins/io/io_file.c` with platform-specific mappings (`_open`/`_read`/`_write`/`_close`/`_lseeki64`/`_commit`/`_unlink`/`_fstat` on Windows).

### New Intrinsics (`lucis::sys`)

- **`read<T>(ptr) → T`**: Typed load from a raw pointer. Emits a direct LLVM `load` instruction — no function call overhead.
- **`write<T>(ptr, val) → void`**: Typed store to a raw pointer. Emits a direct LLVM `store` instruction.
- **`offset<T>(ptr, count) → *T`**: Typed pointer arithmetic via GEP. Advances a pointer by `count` elements of type `T`. The type argument must be a pointer type (e.g. `<*int32>`).
- **`bitcast<T, U>(val) → U`**: Reinterprets the bit pattern of a value as a different type of the same size. Emits LLVM `bitcast`.
- **`assume(cond) → void`**: Optimizer hint asserting a condition is always true (`@llvm.assume`). UB if the condition is false at runtime.
- **`unreachable() → void`**: Marks code as unreachable (`@llvm.unreachable`). UB if control flow reaches it.
- **`size_of<T>() → usize`**: Compile-time type size query (`DataLayout::getTypeStoreSize`).
- **`align_of<T>() → usize`**: Compile-time type alignment query (`DataLayout::getABITypeAlign`).
- **`IntrinsicFunction.returnTypeArgIndex`**: Added field to support multi-type-param intrinsics like `bitcast` where the return type is not the first type argument.

### Memory Safety

- **Fixed**: Memory leak in `io.c` (`lucis_prompt`, `lucis_readLine`, `lucis_readAll`, `lucis_readLines`) — `getline` allocations are now copied into tracked memory so `lucis_freeStr` at scope exit properly frees them.
- **Fixed**: Memory leaks in 14 builtin C modules (~85 functions) that returned untracked heap-allocated strings (`global.c`, `process.c`, `fmt.c`, `path.c`, `regex.c`, `random.c`, `os.c`, `time.c`, `encoding.c`, `crypto.c`, `net.c`, `compress.c`, `vec.c`, `fs.c`). All string payloads now use `lucis_allocString` for proper lifecycle tracking.
- **Fixed**: `getProjectRoot` infinite loop when no `lucis.yaml` is found — the loop now correctly breaks at the filesystem root instead of looping forever.
